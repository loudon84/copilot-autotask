# AutoTask Studio Chrome 会话层真实弹窗修复 PRD

版本：v0.3.1
主题：从“Mock 快速打开”修复为“真实 Chrome / Edge 浏览器弹窗”
项目仓库：https://github.com/loudon84/copilot-autotask
适用模块：客户 SRM、自动化任务、待人工处理、浏览器 Session、Electron IPC

---

## 1. 背景

AutoTask Studio v0.3 已经完成“独立 Chrome Profile + Playwright / CDP 接管”方案的第一轮产品原型。

当前页面已经具备：

```text
客户 SRM 页面：
- 快速打开
- 测试登录
- 查看会话
- 打开 Profile 目录
- 重置登录态

任务列表页面：
- WAITING_HUMAN 快速打开
- HUMAN_OPERATING 重新打开
- 确认已完成
```

但当前实际行为仍然停留在 Mock 阶段：

```text
用户点击【快速打开】
  → 页面提示“已快速打开”
  → 生成一个 mock browser session
  → 但没有真实弹出 Chrome / Edge 窗口
```

本 PRD 目标是修复该问题，让 v0.3.1 实现真实本地浏览器弹窗。

---

## 2. 问题定位

### 2.1 当前主进程已经有真实浏览器启动能力

源码中已经存在：

```text
src/ipc/browser/handlers.ts
```

该文件已经实现：

```text
1. 检测 Chrome / Edge 默认安装路径
2. 分配 CDP 端口
3. 创建 browser session
4. 使用 subprocess spawn 启动浏览器
5. 传入 user-data-dir
6. 传入 remote-debugging-port
7. 传入 --new-window
```

说明 Electron Main Process 侧已经具备真实浏览器启动能力。

---

### 2.2 当前 UI 层仍然调用 Mock 方法

客户 SRM 页面实际点击逻辑仍然是：

```ts
mockApi.openPortalMock(...)
```

该方法只做：

```text
1. 创建前端 BrowserSession 对象
2. 写入 Zustand Store
3. 弹出 toast
4. 不调用 Electron Main
5. 不执行 spawn
6. 不打开真实浏览器
```

因此用户看到 toast：

```text
已快速打开: 客户C 协同门户
Session: browser_sess_xxx
```

但不会出现真实 Chrome / Edge 窗口。

---

### 2.3 待人工任务也仍然调用 Mock 方法

任务列表中的待人工快速打开逻辑仍然是：

```ts
mockApi.openHumanTaskMock(...)
```

该方法同样只做前端状态变更：

```text
WAITING_HUMAN
  → HUMAN_OPERATING
```

不会打开真实浏览器。

---

### 2.4 ProfilePath 当前存在硬编码风险

当前 `src/mock/srm-portals.json` 中 ProfilePath 类似：

```text
C:\Users\demo\AppData\Roaming\AutoTaskStudio\browser-profiles\user_001\portal_001
```

问题：

```text
1. 用户机器上不一定存在 C:\Users\demo
2. 打包后不同用户路径不同
3. macOS / Linux 不兼容
4. 真实 ProfilePath 不应由 Mock JSON 决定
```

因此 v0.3.1 必须改为由 Electron Main 统一生成真实 ProfilePath。

---

## 3. 版本目标

v0.3.1 的目标不是重构完整 Browser Session Service，而是完成“真实弹窗”的最小闭环。

### 3.1 核心目标

```text
1. 客户 SRM 页面点击【快速打开】，真实弹出 Chrome / Edge 窗口。
2. 浏览器使用独立 user-data-dir。
3. CDP 端口仅监听 127.0.0.1。
4. ProfilePath 不再依赖 C:\Users\demo 硬编码。
5. 前端 toast 中显示真实 Main Process 返回的 Session ID。
6. 找不到浏览器时给出明确错误提示。
```

### 3.2 第二阶段目标

```text
1. WAITING_HUMAN 任务点击【快速打开】，真实打开待人工页面。
2. WAITING_HUMAN → HUMAN_OPERATING 状态保留。
3. 用户点击【确认已完成】后进入 SUCCESS_MANUAL。
4. 不自动继续执行后续 RPA 步骤。
```

---

## 4. 不做范围

v0.3.1 不做以下内容：

```text
1. 不接 Playwright Worker 自动执行。
2. 不做 connectOverCDP 接管。
3. 不做登录态自动检测。
4. 不做验证码识别。
5. 不做云端 Browser Session 同步。
6. 不做多 Worker 调度。
7. 不做复杂 Profile Lock。
8. 不做真实 Credential Service。
```

这些内容放到 v0.3.2 / v0.4 继续迭代。

### 6.2.3 新增 ProfilePath 解析函数

```ts
function isMockProfilePath(profilePath?: string): boolean {
  if (!profilePath) return true;

  return (
    profilePath.includes("C:\\Users\\demo") ||
    profilePath.includes("/Users/demo") ||
    profilePath.includes("/home/demo")
  );
}

function resolveProfilePath(input: {
  portalId: string;
  profilePath?: string;
}): string {
  if (!isMockProfilePath(input.profilePath)) {
    return input.profilePath!;
  }

  return path.join(
    app.getPath("userData"),
    "browser-profiles",
    "user_001",
    safePathSegment(input.portalId)
  );
}
```

### 6.2.6 修改 launchBrowser 入参

```ts
async function launchBrowser(input: {
  portalId: string;
  portalName?: string;
  customerName?: string;

  taskId?: string;
  runId?: string;

  profileId?: string;
  profilePath: string;

  targetUrl: string;
  browserType: "chrome" | "edge" | "chromium";
  launchMode?: "manual_open" | "rpa_launch" | "cdp_attach";

  executablePath?: string;
}): Promise<MainBrowserSession> {
  const executable = resolveExecutable(input.browserType, input.executablePath);
  await mkdir(input.profilePath, { recursive: true });

  const cdpPort = allocatePort();
  const sessionId = createSessionId();

  const session: MainBrowserSession = {
    id: sessionId,
    portalId: input.portalId,
    portalName: input.portalName,
    customerName: input.customerName,

    taskId: input.taskId,
    runId: input.runId,

    profileId: input.profileId,
    profilePath: input.profilePath,

    targetUrl: input.targetUrl,
    browserType: input.browserType,
    launchMode: input.launchMode ?? "manual_open",

    cdpPort,
    cdpEndpoint: `http://127.0.0.1:${cdpPort}`,

    status: "STARTING",
    startedAt: now(),
    lastActiveAt: now(),
  };

  const args = [
    `--user-data-dir=${input.profilePath}`,
    "--remote-debugging-address=127.0.0.1",
    `--remote-debugging-port=${cdpPort}`,
    "--no-first-run",
    "--new-window",
    input.targetUrl,
  ];

  const child = spawn(executable, args, {
    detached: true,
    stdio: "ignore",
    windowsHide: false,
  });

  child.unref();

  session.process = child;
  session.pid = child.pid;
  session.status = "OPENED";
  sessions.set(sessionId, session);

  child.on("exit", () => {
    const existing = sessions.get(sessionId);
    if (existing) {
      existing.status = "CLOSED";
      existing.closedAt = now();
      existing.process = undefined;
    }
  });

  child.on("error", (error) => {
    const existing = sessions.get(sessionId);
    if (existing) {
      existing.status = "ERROR";
      existing.errorMessage = error.message;
    }
  });

  return session;
}
```

### 6.2.8 修改 openPortal

```ts
export const openPortal = os
  .input(openPortalInputSchema)
  .handler(async ({ input }) => {
    const existing = Array.from(sessions.values()).find(
      (s) => s.portalId === input.portalId && s.status === "OPENED"
    );

    if (existing) {
      existing.lastActiveAt = now();
      return toPublicSession(existing);
    }

    const profilePath = resolveProfilePath({
      portalId: input.portalId,
      profilePath: input.profilePath,
    });

    const session = await launchBrowser({
      portalId: input.portalId,
      portalName: input.portalName,
      customerName: input.customerName,

      profileId: input.profileId,
      profilePath,

      targetUrl: input.targetUrl,
      browserType: input.browserType,
      executablePath: input.executablePath,
      launchMode: "manual_open",
    });

    return toPublicSession(session);
  });
```

## 9. 安全策略

v0.3.1 必须遵守：

```text
1. 不读取用户默认 Chrome Profile。
2. 不使用用户日常浏览器数据目录。
3. CDP 只监听 127.0.0.1。
4. 不把 remote-debugging-port 暴露到 0.0.0.0。
5. Renderer 不直接调用 child_process。
6. Renderer 只能通过 ORPC / IPC 请求 Main Process。
7. ProfilePath 由 Main Process 兜底生成。
```

## 12. 常见错误与处理

### 12.1 spawn ENOENT

原因：

```text
没有找到 Chrome / Edge 可执行文件。
```

处理：

```text
1. 确认 Chrome / Edge 已安装。
2. 检查 chrome_locator 默认路径。
3. 后续增加浏览器路径手动配置。
```

---

### 12.2 没有弹窗但 toast 成功

原因：

```text
UI 仍在调用 mockApi.openPortalMock。
```

处理：

```text
检查 portal-actions.tsx 中 handleOpen 是否已改为 openPortal。
```

---

### 12.3 ProfilePath 权限错误

原因：

```text
仍使用 C:\Users\demo 或无权限目录。
```

处理：

```text
检查 handlers.ts 是否使用 app.getPath("userData") 生成路径。
```

---

### 12.4 浏览器弹出但不是目标页面

原因：

```text
targetUrl 传入错误，或 quickOpenUrl / loginPageUrl 配置错误。
```

处理：

```text
检查 PortalActions 中 targetUrl 计算逻辑。
```
