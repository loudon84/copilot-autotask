# AutoTask Studio Chrome 会话层方案 PRD

## 独立 Chrome Profile + Playwright / CDP 接管

版本：v0.3 Chrome Session Demo
目标仓库：`loudon84/copilot-autotask`
适用阶段：Mock UI → 本地真实执行能力第一阶段

---

## 1. 背景与目标

AutoTask Studio 当前已经完成 Mock UI 原型，产品定位是面向客户 SRM 自动化作业的桌面工作台，已有信息架构包括工作台、自动化任务、流程模板、RPA 组件库、客户 SRM、运行监控、证据中心和系统设置。

本阶段在现有 Electron 客户端上增加 **本地 Chrome 会话层**，用于解决三个核心问题：

```text
1. 客户 SRM 列表支持【快速打开】
   用户维护客户 SRM 后，可直接在 AutoTask Studio 中打开对应 SRM 系统。

2. WAITING_HUMAN 任务支持【快速打开】
   用户点击后进入对应 SRM 页面完成验证码、MFA、人工上传、人工确认等操作。

3. RPA Worker 复用用户登录态
   用户人工登录过的 SRM Profile 可被 Playwright Worker 复用，减少重复登录和验证码触发。
```

本阶段不做完整云端调度、不做远程浏览器池、不做验证码识别、不做真实 ERP 回写。目标是先在本地完成可演示、可迭代、可真实化的 Chrome 会话闭环。

---

## 2. 方案结论

采用：

```text
独立 Chrome Profile
  + Electron IPC
  + Local Browser Session Service
  + Playwright launchPersistentContext
  + Playwright connectOverCDP
```

核心设计：

```text
一个客户 SRM Portal = 一个独立 Chrome Profile
一个 Chrome Profile 保存该 SRM 的 cookie / localStorage / IndexedDB / 下载设置
用户快速打开和 RPA Worker 共用同一个 Profile
```

推荐优先级：

```text
1. 用户快速打开 SRM：Electron 启动外部 Chrome，使用独立 user-data-dir
2. RPA 自动执行：优先使用 launchPersistentContext 复用该 Profile
3. 用户已打开页面需要接管：使用 connectOverCDP 连接已有 Chrome
4. Profile 被占用：进入 WAITING_HUMAN 或提示用户关闭会话
```

不建议把客户 SRM 直接嵌入 Electron WebView / WebContentsView 作为主方案。原因是 Electron 内置 Chromium Session 与标准 Chrome Profile 不一致，后续 Playwright 复用登录态、企业 SSO、浏览器兼容性都会更复杂。

## 5. 核心业务规则

### 5.1 Profile 规则

```text
1. 每个 SRM Portal 独立 Profile。
2. Profile 不能使用用户日常 Chrome 默认目录。
3. Profile 路径由系统生成，不允许用户手工随意输入。
4. 同一时间一个 Profile 只能被一个执行会话占用。
5. 用户快速打开和 RPA 执行共用同一个 Profile，但必须通过锁管理避免并发冲突。
```

Profile 路径：

```text
Windows:
%APPDATA%/AutoTaskStudio/browser-profiles/{userId}/{portalId}

macOS:
~/Library/Application Support/AutoTaskStudio/browser-profiles/{userId}/{portalId}

Linux:
~/.config/AutoTaskStudio/browser-profiles/{userId}/{portalId}
```

### 5.2 Chrome 启动规则

```text
1. remote-debugging-address 固定为 127.0.0.1。
2. remote-debugging-port 使用随机空闲端口。
3. 每个 browserSession 保存 pid、cdpEndpoint、portalId、taskId。
4. Chrome 异常退出时，Session 状态更新为 CLOSED。
5. 如果 Profile 已被其他 Session 占用，禁止重复启动。
```

启动参数：

```bash
chrome \
  --user-data-dir="{profilePath}" \
  --remote-debugging-address=127.0.0.1 \
  --remote-debugging-port={port} \
  --no-first-run \
  --new-window "{targetUrl}"
```

### 5.3 WAITING_HUMAN 规则

你现在明确要求：“待人工”只需要引导用户进入页面完成操作，完成后在任务上点击确认即可，不需要后台自动执行任务的下个阶段。

因此状态机改为：

```text
READY
  ↓
RUNNING
  ├─ SUCCESS
  ├─ FAILED
  └─ WAITING_HUMAN
        ↓ 用户点击【快速打开】
      HUMAN_OPERATING
        ↓ 用户点击【确认已完成】
      SUCCESS_MANUAL
```

解释：

```text
WAITING_HUMAN：
系统已识别需要人工处理，但用户还没有打开页面。

HUMAN_OPERATING：
用户已经点击快速打开，正在 Chrome 中处理。

SUCCESS_MANUAL：
用户确认已完成，本任务闭环。系统不再继续自动执行后续步骤。
```

### 9.2 browser-context.ts

```ts
import { contextBridge, ipcRenderer } from 'electron'
import { BROWSER_CHANNELS } from './browser-channels'

export function exposeBrowserContext() {
  contextBridge.exposeInMainWorld('autotaskBrowser', {
    openPortal: (input: { portalId: string; targetUrl?: string }) =>
      ipcRenderer.invoke(BROWSER_CHANNELS.OPEN_PORTAL, input),

    openHumanTask: (input: { taskId: string }) =>
      ipcRenderer.invoke(BROWSER_CHANNELS.OPEN_HUMAN_TASK, input),

    listSessions: () =>
      ipcRenderer.invoke(BROWSER_CHANNELS.LIST_SESSIONS),

    closeSession: (input: { sessionId: string }) =>
```

## 11. RPA Worker 设计

当前 AutoTask Studio 的技术方案中，RPA Worker 已定义为 Python + Playwright，负责启动浏览器、管理 Browser Context、执行流程 Step、截图、Trace、下载文件和上报运行事件。

本阶段新增 Browser Adapter：

```text
apps/rpa-worker/runtime/browser/
├─ browser_profile_resolver.py
├─ playwright_chrome_adapter.py
├─ cdp_attach_adapter.py
├─ login_state_checker.py
└─ browser_lock_client.py
```

### 11.1 执行策略

```text
RPA 执行前：
1. 根据 task.portalId 请求 /api/local/browser/resolve-for-rpa
2. 如果返回 launch_persistent_context：
   使用 Playwright launchPersistentContext 启动独立 Profile
3. 如果返回 connect_over_cdp：
   使用 Playwright connectOverCDP 接管已打开 Chrome
4. 执行登录态检查
5. 已登录则继续执行
6. 未登录则按配置自动登录或进入 WAITING_HUMAN
7. 结束后释放 Profile Lock
```

### 11.2 launchPersistentContext 示例

```ts
const context = await chromium.launchPersistentContext(profilePath, {
  channel: 'chrome',
  headless: false,
  acceptDownloads: true,
  downloadsPath: artifactDownloadDir,
  tracesDir: artifactTraceDir,
})

const page = context.pages()[0] ?? await context.newPage()
await page.goto(task.targetUrl)
```

### 11.3 connectOverCDP 示例

```ts
const browser = await chromium.connectOverCDP(cdpEndpoint)
const context = browser.contexts()[0]
const page = context.pages()[0] ?? await context.newPage()
await page.goto(task.targetUrl)
```

## 15. 风险与约束

| 风险                 | 说明                      | 处理                               |
| ------------------ | ----------------------- | -------------------------------- |
| Chrome Profile 并发锁 | 同一个 Profile 不能被多个进程同时写入 | 增加 Profile Lock                  |
| 企业 SRM SSO 复杂      | 某些客户 SRM 依赖企业证书、插件、MFA  | 优先人工快速打开，RPA 只复用登录态              |
| CDP 端口安全           | CDP 可控制浏览器              | 仅监听 127.0.0.1，端口随机，Session 运行时可见 |
| 登录态失效              | Cookie 过期后 RPA 无法执行     | 转 WAITING_HUMAN                  |
| 用户误删 Profile       | 登录态丢失                   | 提供重置确认弹窗与审计                      |
| 浏览器路径不一致           | 不同系统 Chrome 安装路径不同      | Chrome Locator 自动检测 + 手动配置       |
