# PRD 1：AutoTask 前端数据接入与登录复用方案

## 1. 项目定位

AutoTask 项目保持独立客户端产品定位：

```text
AutoTask Studio Client
= Electron + React + shadcn/ui 桌面工作台
= 任务操作台
= 客户 / 供应商 Portal 配置台
= RPA 任务状态查看台
= 人工接管入口
= 本地 Web 工作区 / 快速打开入口
```

AutoTask 不直接实现用户体系、不直接维护组织角色、不直接保存后端任务数据库。用户登录、Token、组织身份由 `nodeskclaw-backend` 提供；任务、Portal、Workflow、Run、Artifact 等 AutoTask 业务数据由新项目 `nodeskclaw-task` 提供。

## 2. 当前状态

AutoTask mock 版本已完成用户操作验证。既有 Mock PRD 已明确要求：

```text
1. 页面不直接 import JSON
2. 页面统一调用 mockApi
3. mockApi 返回 Promise，模拟接口请求
4. 后续真实 API 替换时，只替换 services 层
```

因此本阶段的目标不是重写页面，而是将 `src/services/mock-api.ts` 替换为真实 API service，并将页面数据源切换到 TanStack Query。

## 3. 本阶段目标

实现 AutoTask 从 mock 数据到真实前后端数据对接：

```text
1. 复用 copilot-desktop / ai-os-desktop 的登录模式
2. 使用 AutoTask 自己的 shadcn UI 组件重写登录页
3. 接入 nodeskclaw-backend 认证接口
4. 接入 nodeskclaw-task 业务接口
5. 用真实 API 替换 mock 数据
6. 保留本地 mock fallback，方便演示和离线开发
7. 为后续 RPA Local Agent / Server Worker 接入预留接口
```

## 4. 登录复用策略

### 4.1 只复用登录逻辑，不复用视觉 UI

`ai-os-desktop` 的 `LoginScreen.tsx` 采用以下模式：

```text
1. 读取 endpoint 配置
2. 读取上一次登录账号
3. 初始化时调用 window.desktopAuth.getState()
4. 已登录时执行 bootstrap
5. 未登录时显示登录表单
6. 提交时保存 endpointConfig
7. 调用 window.desktopAuth.login()
8. 保存最后登录账号
9. setAuthState()
10. runBootstrap()
11. 登录成功后进入主应用
```

该模式可复用，但 UI 不直接复制。AutoTask 应用自己的 Card、Input、Button、Form、Alert、Tabs 等 shadcn 组件重新实现 `AutoTaskLoginScreen`。`LoginScreen.tsx` 当前确实通过 `window.desktopAuth.getState()`、`window.desktopAuth.login()`、`saveLastLoginEmail()`、`runBootstrap()` 组织登录生命周期。

### 4.2 AutoTask 登录模块结构

建议新增：

```text
src/renderer/src/modules/auth/
├─ AutoTaskAuthProvider.tsx
├─ AutoTaskLoginScreen.tsx
├─ components/
│  ├─ LoginForm.tsx
│  ├─ EndpointConfigPanel.tsx
│  └─ BootstrapScreen.tsx
├─ last-login-email.ts
└─ auth.types.ts
```

Electron 主进程新增：

```text

├─ auth-client.ts
├─ auth-ipc.ts
├─ auth-endpoint-config-store.ts
├─ token-store.ts
├─ token-header-injector.ts
└─ nodeskclaw-auth-response.ts
```

Preload 新增：

```text

```

Renderer 暴露：

```ts

window.autotaskAuth.saveEndpointConfig()
window.autotaskAuth.login()
window.autotaskAuth.logout()
window.autotaskAuth.refresh()
```

注意：AutoTask 不应直接把 accessToken 长期放在 renderer 状态里。`ai-os-desktop` 的 token 存储逻辑在主进程中优先使用 `keytar`，失败后使用 Electron `safeStorage` 加密文件，再降级为内存会话；这套模式适合直接复用。

## 5. Endpoint 配置

AutoTask 需要两个后端地址：

```ts
export interface AutoTaskEndpointConfig {
  authBackendUrl: string
  authPrefix: string
  taskBackendUrl: string
  taskPrefix: string
  aiosHomeUrl?: string
}
```

默认开发配置：

```ts
export const defaultAutoTaskEndpointConfig = {
  authBackendUrl: 'http://127.0.0.1:4510',
  authPrefix: '/api/v1/auth',
  taskBackendUrl: 'http://127.0.0.1:4520',
  taskPrefix: '/api/v1/autotask',
  aiosHomeUrl: 'http://127.0.0.1:4517',
}
```

`ai-os-desktop` 已有 endpoint 规范：`AuthEndpointConfig` 包含 `backendUrl / authPrefix / aiosHomeUrl`，并通过 `buildAuthUrl(config, path)` 拼接 `login / account-login / refresh / me / logout` 等认证路径。

## 6. 认证流程

### 6.1 登录流程

```text
用户打开 AutoTask
  ↓
AutoTaskAuthProvider 初始化
  ↓
window.autotaskAuth.getState()
  ↓
已有有效 session？
  ├─ 是：进入 AppShell
  └─ 否：进入 AutoTaskLoginScreen
        ↓
      用户输入账号、密码、endpoint
        ↓
      window.autotaskAuth.saveEndpointConfig()
        ↓
      window.autotaskAuth.login()
        ↓
      POST nodeskclaw-backend /api/v1/auth/account-login
        ↓
      GET nodeskclaw-backend /api/v1/auth/me
        ↓
      主进程安全保存 token
        ↓
      Renderer 只拿 public auth state
        ↓
      进入 AutoTask AppShell
```

`ai-os-desktop` 的 HTTP AuthClient 登录时会请求 `account-login`，成功后继续请求 `me`，并将返回的节点身份映射为本地 StoredAuthSession。

### 6.2 Token 刷新

```text
接口 401
  ↓
window.autotaskAuth.refresh()
  ↓
POST /api/v1/auth/refresh
  ↓
刷新成功：重放请求
刷新失败：清理 session，跳转登录页
```

nodeskclaw-backend 已提供 `/api/v1/auth/refresh`，其响应包含 `access_token / refresh_token / token_type / expires_in`。

### 6.3 登出

```text
用户点击退出
  ↓
window.autotaskAuth.logout()
  ↓
POST /api/v1/auth/logout
  ↓
清除本地 token-store
  ↓
清理本地 Web 工作区 session
  ↓
返回登录页
```

## 7. API 调用方式

推荐优先采用安全模式：

```text
Renderer TanStack Query
  ↓
window.autotaskApi.request(...)
  ↓
Electron Main Process
  ↓
读取 token-store
  ↓
向 nodeskclaw-task 发起 HTTP 请求并附加 Authorization
  ↓
返回 JSON 给 Renderer
```

不推荐第一期就让 renderer 直接持有 accessToken。原因：

```text
1. 与 ai-os-desktop 的安全模型一致
2. Token 不暴露给 React 页面
3. 便于统一 refresh / retry / 401 处理
4. 便于后续接本地 Local Agent、WebContentsView、RPA Profile
```

如果为了开发速度，可以临时开放 `window.autotaskAuth.getAccessToken()`，但应标记为临时方案，并在正式版本移除。

```text
src/features/tasks/api/
├─ use-tasks.ts

├─ use-create-task.ts
├─ use-start-task.ts
├─ use-cancel-task.ts
└─ use-retry-task.ts

├─ use-workflow-templates.ts
└─ use-workflow-template.ts

src/features/portal-accounts/api/
├─ use-portal-accounts.ts
└─ use-portal-account.ts

src/features/runs/api/
├─ use-runs.ts
├─ use-run.ts
└─ use-run-events.ts
```

示例：

```ts
export function useTasks(params: TaskListParams) {
  return useQuery({
    queryKey: queryKeys.tasks.list(params),
    queryFn: () => autotaskApi.tasks.list(params),
  })
}
```

shadcn-admin 的结构本身就是 feature slice 模式，数据表格也推荐由 feature 内部 columns + hooks 提供数据；其文档明确指出，当连接真实 API 时，应将 `src/features/*/data/*.ts` 的静态数组替换为 TanStack Query 的 `useQuery / useMutation` hooks。

## 12. 实时事件接入

第一期使用轮询：

```text
任务详情页：每 2 秒 refetch task + run events
运行详情页：每 1 秒 refetch run events
Dashboard：每 5 秒 refetch summary
```

第二期接 SSE：

```text
GET /api/v1/autotask/events?token=<sse_token>
```

nodeskclaw-backend 已有 SSE token 模式，例如 workspace events 使用 query param JWT，并提供短时效 SSE token。AutoTask 可复用该模式，但由 nodeskclaw-task 单独签发 AutoTask SSE token。

Electron IPC 仍沿用 `electron-shadcn` 的安全边界：Renderer 不直接访问 Node 能力，新增能力通过 preload 暴露 `window.*` API。该模板的核心约束是 Main / Preload / Renderer 三进程结构，并要求新增 IPC 能力走 channels / listeners / context 的模式。

## 14. 实施步骤

### Phase A：登录接入

```text
1. 新增 AutoTaskAuthProvider
2. 新增 AutoTaskLoginScreen
3. 新增 auth IPC
4. 新增 token-store
5. 接入 nodeskclaw-backend /api/v1/auth/account-login
6. 接入 /api/v1/auth/me
7. 接入 /api/v1/auth/refresh
8. 接入 /api/v1/auth/logout
9. 登录后进入现有 AppShell
```

验收：

```text
用户可以使用 nodeskclaw 账号登录 AutoTask
刷新应用后仍保持登录
Token 过期后可以 refresh
refresh 失败回到登录页
Renderer 不直接持久化 accessToken
```

### Phase B：API Client 接入

```text
1. 新增 endpoint-config
2. 新增 autotaskApi.request IPC
3. 新增 remote-api
4. 新增 mock / remote 切换
5. 新增 query-keys
6. 新增 DTO mapper
```

验收：

```text
VITE_AUTOTASK_API_MODE=mock 时仍使用本地 JSON
VITE_AUTOTASK_API_MODE=remote 时访问 nodeskclaw-task
API 401 自动 refresh
API 403 显示权限不足
API 500 显示错误 toast
```

### Phase C：页面替换 mock

```text
1. Dashboard 替换 dashboard.json
2. Tasks 替换 tasks.json
3. Task Detail 替换 task-runs.json / audit-logs.json
4. Workflows 替换 workflow-templates.json
5. SRM Portals 替换 srm-portals.json
6. Runs 替换 task-runs.json
7. Artifacts 替换 artifacts.json
8. Workers 替换 workers.json
```

验收：

```text
页面不再直接读取 JSON
所有数据来自 query hook
新增任务可以写入后端
任务状态操作可以调用后端 mutation
详情页刷新后数据不丢失
```

### Phase D：任务操作接入

```text
1. 创建任务
2. 编辑任务
3. 执行任务
4. 取消任务
5. 重试任务
6. 人工接管
7. 人工确认完成

```

验收：

```text
点击执行后后端任务状态变为 QUEUED / RUNNING
点击取消后任务变为 CANCELLED
点击重试后任务重新进入 QUEUED
WAITING_HUMAN 任务可打开快速处理入口
Artifact 可查看 metadata 和下载链接
```

## 15. 前端验收标准

```text
1. 登录逻辑与 nodeskclaw-backend 打通
2. AutoTask 页面保持现有 UI 风格
3. mock 数据可一键切换 remote API
4. 所有主页面完成真实 API 替换
5. 任务创建、任务列表、任务详情、运行记录、Artifact 全链路可用
6. Token 存储在 Electron Main 侧
7. Renderer 只持有 public auth state
```
