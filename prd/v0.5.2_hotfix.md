目标模块：Dashboard / Remote API / DTO Mapper

1. 背景

当前 AutoTask 前端登录 auth 成功后，切换到远程 nodeskclaw-task 接口时，Dashboard 页面出现：

Cannot read properties of undefined (reading 'map')

报错位置在：

dashboard?.taskTypeDistribution.map(...)

当前源码只保护了 dashboard 是否为空，但没有保护 dashboard.taskTypeDistribution 是否为空。

前端 Dashboard 类型要求数据结构为：

interface DashboardData {
  stats: DashboardStats;
  taskTypeDistribution: TaskTypeDistribution[];
}

其中 stats 包含 pending、running、waitingHuman、failed、completedToday、successRate。

但后端当前返回的是：

{
  "code": 0,
  "message": "success",
  "data": {
    "todayTotal": 0,
    "ready": 0,
    "running": 0,
    "waitingHuman": 0,
    "failed": 0,
    "success": 0,
    "successRate": 0.0,
    "onlineWorkers": 0
  }
}

该结构既没有 stats，也没有 taskTypeDistribution，导致页面渲染崩溃。

2. 问题根因
2.1 Dashboard 页面缺少防御式渲染

当前代码：

const stats = dashboard?.stats;

{dashboard?.taskTypeDistribution.map((item) => (...))}

如果 dashboard 存在但 taskTypeDistribution 不存在，会直接崩溃。

2.2 DTO Mapper 没有解包统一响应结构

当前 mapItemResponse 只是调用 mapKeysToCamel，不会从 { code, data } 中取出 data。

当前 mapListResponse 只支持：

Array
{ items: [] }
{ data: [] }

不支持：

{
  "code": 0,
  "data": {
    "items": []
  }
}

也不支持 data 不是数组时的明确保护。

2.3 远程 API 与 UI DTO 没有专门适配层

当前远程 Dashboard 直接：

return mapItemResponse<DashboardData>(data);

对应接口是：

GET /dashboard/summary

源码：

但真实后端返回的是扁平统计字段，不是 DashboardData。

2.4 /tasks 接口疑似返回了统计对象

前端远程任务列表调用的是：

GET /tasks

源码：

如果 /api/v1/autotask/tasks 返回的是统计对象，则该接口语义错误。前端期望它返回任务列表，而不是 Dashboard 统计。

3. 目标

本次前端修复目标：

1. Dashboard 不再因 taskTypeDistribution 缺失而白屏。
2. 前端支持后端统一响应 envelope：{ code, message, data }。
3. Dashboard 远程数据增加专门 DTO 适配层。
4. /tasks、/workers、/runs 等列表接口统一通过 mapListResponse 解包。
5. 保持 Mock 模式不受影响。
6. 保持现有 UI 页面结构不大改。
4. 非目标
1. 不重构整个前端架构。
2. 不修改登录流程。
3. 不修改 Electron IPC 主链路。
4. 不新增复杂状态管理。
5. 不要求后端立即完全改成前端字段名。

5.2 DTO Mapper 增加 envelope 解包

修改文件：

src/services/dto-mappers.ts

新增：

type ApiEnvelope = {
  code?: number;
  errorCode?: string | null;
  error_code?: string | null;
  message?: string | null;
  messageKey?: string | null;
  message_key?: string | null;
  data?: unknown;
};

function isApiEnvelope(data: unknown): data is ApiEnvelope {
  return (
    Boolean(data) &&
    typeof data === "object" &&
    "data" in data &&
    ("code" in data || "message" in data || "error_code" in data)
  );
}

function unwrapEnvelope(data: unknown): unknown {
  if (isApiEnvelope(data)) {
    return data.data;
  }

  return data;
}

调整：

export function mapListResponse<T>(data: unknown): T[] {
  const payload = unwrapEnvelope(data);

  if (Array.isArray(payload)) {
    return payload.map((item) => mapKeysToCamel<T>(item));
  }

  if (payload && typeof payload === "object") {
    const wrapped = payload as { items?: unknown[]; data?: unknown[] };
    const items = wrapped.items ?? wrapped.data ?? [];

    return Array.isArray(items)
      ? items.map((item) => mapKeysToCamel<T>(item))
      : [];
  }

  return [];
}

export function mapItemResponse<T>(data: unknown): T {
  return mapKeysToCamel<T>(unwrapEnvelope(data));
}

5.4 API 错误信息保留真实后端错误

当前 renderer 层 requestAutotaskApi 捕获错误后固定变成 ApiClientError(message, 500)，会丢失后端真实 status。

建议改为：

export async function requestAutotaskApi<T>(
  input: AutotaskApiRequestInput
): Promise<T> {
  try {
    return (await ipc.client.autotaskApi.request({
      method: input.method,
      path: input.path,
      body: input.body,
      query: normalizeQuery(input.query),
    })) as T;
  } catch (err) {
    if (err instanceof ApiClientError) {
      throw err;
    }

    const message = err instanceof Error ? err.message : "API 请求失败";
    throw new ApiClientError(message, 500);
  }
}

更进一步可以让 ORPC 透传 AutotaskApiError.status，但不是本次 P0 必须项。
